# 📐 在线桌游“猜数字”项目技术方案

## 一、游戏核心机制（最终确认）

### 1.1 牌组构成
- **总牌数**：60张，5种颜色（红、蓝、绿、橙、粉）各12张。
- **数字规律**：颜色与数字绑定循环（红1、蓝2、绿3、橙4、粉5、红6……粉60）。
- **点数规律**：`点数 = floor((数字 - 1) / 5) + 1`，范围 1~12。

### 1.2 初始发牌
- 每人从牌堆抽取5张**颜色不同**的牌，按**数字从小到大**在面前排列。
- **可见性**：玩家仅知自己牌的颜色（数字未知），但可看到其他玩家完整的颜色+数字。

### 1.3 公开牌区（公共牌堆）
- 开局亮出5张颜色各异的公牌。
- 每轮次，当前玩家**选择一种颜色**（系统展示该颜色剩余数量），从该颜色剩余牌中**按预洗牌顺序**抽取一张加入公牌区（此时公牌变为6张）。
- **判定后**：被选中的公牌**移入弃牌堆**，公牌区恢复为5张。

### 1.4 判定机制（二选一）
- **1类判定（位置）**：选定一张公牌，系统返回该数字位于当前玩家手牌（5张）中的位置（索引 0~5：0=小于最小，5=大于最大，1~4=介于第N张与第N+1张之间）。
- **2类判定（点数）**：选定一张公牌 + 指定自己手牌中的某颜色，系统返回两者点数是否相同。

### 1.5 胜负、退出与平局
- **随时提交猜测**：任何存活玩家可在任何时候提交自己5张牌的完整数字。
  - **正确**：整场游戏结束，该玩家获胜。
  - **错误**：该玩家手牌**全部数字公开**，标记出局（转为旁观者），游戏继续。
- **平局**：若牌堆（所有颜色队列）耗尽且无玩家猜对，**强制平局，无胜者**。

## 二、后端核心架构

### 2.1 状态机（含暂停与高优先级中断）
| 状态 | 描述 | 触发条件 |
| :--- | :--- | :--- |
| `WAITING` | 等待大厅 | 房间创建，满4人前 |
| `PLAYING` | 游戏进行中 | 房主点击开始，且4人在线 |
| `GAME_OVER` | 已结束 | 猜对成功 / 平局 |

**`PLAYING` 子状态机**：
1. **`DRAW_PHASE`**：系统自动抽取一张公牌（当前玩家选色）。
2. **`ACTION_PHASE`**：等待当前玩家执行判定，或房主强制跳过。
3. **`JUDGE_RESULT`**：结算判定，记录线索，切换下一位玩家。
4. **`PAUSED`（中断态）**：仅在当前操作玩家断线时触发，**冻结所有倒计时**。

**中断机制**：在任何 `PLAYING` 子状态下，若收到任意存活玩家的 `SubmitGuess`，**立即暂停**当前流程，校验结果并广播，然后跳转 `GAME_OVER` 或继续（若猜错）。

### 2.2 牌堆随机策略（种子预洗牌）
- 房间创建时生成一个**随机种子（Seed）**，固化于房间对象。
- 基于该种子，将60张牌按颜色分组（5组各12张），组内乱序。
- 玩家选色抽牌时，从该颜色队列**依次弹出（Pop）**下一张。
- **优势**：结果可复现、可审计，杜绝网络波动引发的随机争议。

### 2.3 数据模型与同步策略（核心简化）
**关键决策：后端不搞差异化视图，只维护一份绝对真理。**
- 后端内存中保存所有玩家的**真实完整手牌**（含数字）、公牌、弃牌堆。
- 每当状态变化，后端只序列化**一份全量数据**，通过 `broadcast()` 一次性发送给所有玩家。
- **前端负责过滤**：自己在渲染时，对自己手牌的数字字段做隐藏（显示为`?`），对其他玩家的手牌正常显示数字。

## 三、网络通信协议（WebSocket）

### 3.1 消息格式
统一 JSON，区分 `cmd`（客户端请求）与 `event`（服务端推送）。

### 3.2 C2S（客户端 → 服务端）指令
| 指令 | 携带参数 | 权限 |
| :--- | :--- | :--- |
| `JoinRoom` | 房间号, 玩家名 | 全员 |
| `Reconnect` | 房间号, 玩家ID, `accessToken` | 断线玩家 |
| `StartGame` | 无 | 仅房主 |
| `SelectColorToDraw` | 颜色（红/蓝/绿/橙/粉） | 当前操作玩家 |
| `JudgeByPosition` | 公牌ID | 当前操作玩家 |
| `JudgeByPoint` | 公牌ID, 自己手牌颜色 | 当前操作玩家 |
| `SkipTurn` | 无 | 仅房主 |
| `SubmitGuess` | 5个数字（按颜色顺序） | 任意存活玩家 |
| `KickPlayer` | 玩家ID | 仅房主（PAUSED态） |

### 3.3 S2C（服务端 → 客户端）事件（统一广播）
后端一律**广播全量数据**，前端按需取用。
- **`Event_StateUpdate`**：游戏过程中每次状态变更（抽牌、判定、翻牌）触发，携带**完整游戏快照**（结构见下文）。
- **`Event_SyncFullState`**：专门用于重连响应，携带完整快照（结构与 `Event_StateUpdate` 一致）。
- **`Event_GuessResult`**：提交猜测后的校验结果（正确/错误，错误时附带正确数字）。
- **`Event_PlayerDisconnected`**：广播断线事件（携带是否触发暂停标志）。
- **`Event_GameOver`**：终局广播（赢家/平局）。

> **全量快照数据结构（核心字段）**：
> ```json
> {
>   "roomId": "ABC123",
>   "gamePhase": "PLAYING",
>   "subPhase": "ACTION_PHASE",
>   "roundNumber": 5,
>   "currentTurnPlayerId": "P2",
>   "deckRemainingCount": { "红": 3, "蓝": 5, ... },
>   "publicCards": [ {"id":"pub_1","color":"红","number":6,"isSelected":false} ],
>   "players": {
>     "P1": { "playerName":"Alice", "isAlive":true, "isOnline":true, "hand": [{"color":"红","number":11}, ...], "clues":[...] },
>     "P2": { ... }  // 所有玩家的手牌均包含真实数字
>   },
>   "pendingJudgement": { "selectedPublicCardId":"pub_2", "judgeType":"point" },
>   "gameOverInfo": null
> }
> ```

## 四、身份认证与重连机制

### 4.1 登录身份令牌
- 玩家通过 `JoinRoom` 成功加入时，后端下发 `accessToken`（UUID或JWT，有效期仅限本局）。
- 前端存储于 `sessionStorage`。

### 4.2 重连身份验证流程
1. 断线玩家发起 `Reconnect`，携带 `roomId`, `playerId`, `accessToken`。
2. 后端校验链（**三者缺一不可**）：
   - 房间是否存在；
   - `playerId` 是否在该房间玩家列表中；
   - 内存中的 `accessToken` 是否与请求一致。
3. **校验通过**：踢掉该身份的旧连接（若有），下发 `Event_SyncFullState`，恢复游戏。
4. **校验失败**：返回拒绝并关闭连接，防止冒充。

## 五、前端架构设计（Vue 3 + Pinia）

### 5.1 三层结构
- **数据层（Store）**：
  - 存储 `Event_StateUpdate` 中的全量快照，不做任何增删改。
  - 独立存储当前玩家ID（`myPlayerId`），用于UI判断“哪些牌是我自己的”。
- **计算层（Composables）**：
  - `useMyHand`：从Store中提取自己的手牌，将 `number` 字段替换为 `null` 或 `?`（仅用于渲染）。
  - `useDeduction`：根据 `clues` 数组，推导候选范围、已确定数字。
  - `useTableFilter`：结合 `revealedNumbers`（已出现数字）和用户手动搁置，计算备选区灰显状态。
- **展示层（Components）**：
  - 纯UI组件，**绝不直接访问原始Store中的数字字段**，一律通过 `useMyHand` 等计算层取数，确保数字绝不会因模板误渲染而泄露。

### 5.2 布局与组件规划（左右比 3:1）
- **左侧主区**：
  - **公共牌区**：5/6张牌横排，支持点击选中/取消（高亮光效）。
  - **其他玩家手牌**：逐行展示，大卡片形式，显示完整数字。
  - **自己的牌区**：5张牌横排，数字隐藏（显示`?`），包含**便签条**（点数线索）与**间隙槽**（位置线索）。
  - **备选区表格**：5行×12列，支持PC右键/触摸滑动手动搁置，系统自动标灰已出现数字。
- **右侧操作栏**：
  - 轮次、计时器、当前操作玩家。
  - 动态按钮区：选色抽牌、判定按钮、提交猜测、房主跳过/踢出。

## 六、交互操作规范（PC + 触摸共存）

### 6.1 公共牌选择（判定选牌）
- **PC**：悬浮放大上浮；单击高亮（边缘光效）；再次单击取消选中；单击其他牌切换选中。
- **触摸**：点击完成“放大上浮→复原”动画，同时高亮；再次点击取消选中。

### 6.2 备选区表格
- **PC**：左键点击提交猜测（填入自己牌区）；右键点击搁置/激活（标灰/恢复）。
- **触摸**：长按数字（触感反馈）→ 上滑提交，下滑搁置。
- **灰度区分**：系统自动标灰（已出现数字）与用户手动搁置使用不同纹理/透明度。

### 6.3 自己牌区
- 仅通过备选区提交或系统判定结果自动填入，不提供直接键盘输入（防止误触）。

## 七、断线、暂停与房主特权

### 7.1 断线处理
- **非当前操作玩家断线**：游戏正常进行，该玩家标记离线，重连后通过 `Event_SyncFullState` 追回状态。
- **当前操作玩家断线**：
  1. 后端自动进入 `PAUSED` 状态，广播暂停事件。
  2. 其他玩家界面遮罩，显示“等待重连...”。
  3. 重连后自动解除暂停，恢复操作权。
  4. 房主可在暂停界面点击“强制踢出”，该玩家判负出局（手牌全亮），游戏继续。

### 7.2 房主特权
- **强制跳过**：在任何 `ACTION_PHASE` 点击跳过，当前玩家本次判定作废，直接进入下一人轮次。
- **强制踢出**：仅在 `PAUSED` 态可用，等同于猜错惩罚。

## 八、打包、部署与运维

### 8.1 内网穿透
- 使用 **SakuraFrp** 等平台进行 TCP/WebSocket 穿透，暴露本地服务端口（如 `8000`）。

### 8.2 一键启停（Python + Vue）
- **后端**：Python FastAPI，同时承担 WebSocket 服务端和静态文件服务器。
- **前端**：`npm run build` 生成 `dist` 目录，挂载至 FastAPI 根路由。
- **启动脚本（`start.py`）**：
  1. 检查 Python 依赖与前端 `dist` 目录（若缺失则自动 `npm install && npm run build`）。
  2. 启动 Uvicorn（`--host 0.0.0.0 --port 8000`）。
  3. 自动打开浏览器访问 `http://localhost:8000`。
- **停止**：终端 `Ctrl+C` 或调用管理接口 `/admin/shutdown`（密钥保护）。

### 8.3 数据持久化
- 游戏全内存运行，房间对象在终局后销毁。可选将事件流落盘为 JSON 日志供复盘。

## 九、边界与异常处理汇总

| 场景 | 方案 |
| :--- | :--- |
| 牌堆所有颜色耗尽 | 立即触发平局，`GAME_OVER` 无胜者 |
| 非当前回合玩家提交猜测 | 允许，正确结束游戏，错误则惩罚出局（中断当前流程） |
| 重连 Token 错误或不匹配 | 拒绝连接，关闭 Socket |
| 玩家在断线期间被房主踢出 | 重连时返回 `Rejected`，前端跳转大厅 |
| 触摸长按误触 | 设置触发阈值（300ms），并给予触感/视觉反馈 |

## 十、扩展预留
- **观战模式**：当前出局玩家即为旁观者，未来可支持无身份观战。
- **AI 陪玩**：计算层（`useDeduction`）可独立为 AI 决策引擎。
- **回放系统**：事件流落盘即可实现战局回放。
